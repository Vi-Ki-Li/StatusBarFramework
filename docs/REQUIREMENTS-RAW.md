## 通用状态栏框架脚本

最终的成品简单来说应该是：一个高度可自定制、支持显示多角色状态数据、支持双端（PC端移动端）良好显示的"通用状态栏框架"；支持"即插即用"，不需要额外导入其他东西（例如可能需要的世界书应该由脚本自动创建管理，可能需要的隐藏AI输出的正则也由脚本管理）；允许用户填入api key调用AI agent来直接编写想要的前端（提供一些提示词，会包括框架脚本可能的api接口、编写注意事项、实战案例等等，允许用户发送和修改提示词），又或者重写数据变量。

核心理念：让不会前端编写的用户也可以为作者没有设置前端的角色卡自由创作前端游玩，所以**必须提供绝对的自定义自由**。系统的所有行为，包括数据如何解析、状态栏/管理器UI渲染/布局怎么样、状态栏脚本等等其他我还没想到的东西，都可以由用户在状态栏管理器中自定义，绝非脚本硬编码（除了必要的）。

### 美学风格

- 专业，信息清晰，允许用户自己根据想要的主题来实现不同风格（例如想要武侠风，可能会想要将状态栏的背景图变成书卷模样；想要海贼王风格，可能会希望角色选择栏（暂定名）做成悬赏海报+显示头像）
- 内置完善且视觉效果优雅的现代简洁风深浅色主题（所有内置的UI组件都应该可切换深浅色）
  - 所有内置组件基于统一的基础组件库构建？需要确保风格一致，并允许用户通过"样式工坊"（暂定名）进行深度定制
  - 可以有自己独立的、少量的CSS，但仅用于其特有的布局和样式，覆盖或补充基础组件库？

### 补充预期成品会包括/能做到/游玩体验

- 一个在聊天界面展示的状态栏 + 一个通过点击在#extensionMenu拓展菜单的按钮作为浮窗打开的状态栏管理器
  - 可以**由用户自定义**（通过脚本？）游戏式交互并游玩，例如最常见的点击事件
    - e.g., RPG主题中，点击在"装备"子面板下的"盔甲"后自动将"装备【盔甲】"填写在输入框或用户指定textarea；在限定时间内高频率点击在"攻击行动"子面板下的"大力劈向xxx"时显示悬浮的进度条，连续点击进度条为攻击充能，限定时间过后自动将伤害计算结果显示在状态栏（xxx如果有被记录"生命值"则"生命值"自动扣减），同时将结果（可能是"{{user}}大力劈向xxx，造成x点伤害，xxx生命值从xx点下降至xx点"）直接发送给AI）
  - 可以有浮窗展示角色关系图谱，例如星型拓扑/网状拓扑/组织架构图等等
  - 让聊天中AI对数值变化更敏感，允许多元描写，由发送不同叙事风格指导提示词给AI来达成？
  - 接洽MVU/JSON Patch数据解析，例如根据[initvar]的json自动构建数据项解析
  - 用户自定义数据
  - 用户一键导出做好的状态栏

## 状态栏

- 功能: 嵌入在聊天消息中，作为信息的核心展示窗口。
- 核心要求:
  1. 完全可定制渲染:
      - 条目外观: 渲染逻辑必须由用户在"样式工坊"中定义的"样式单元"驱动。（状态栏的外观不固定，由用户应用的HTML+css模板动态生成）
      - 布局排版: 用户必须能在"布局编排器"中，通过可视化拖拽的方式，完全自定义状态栏内所有条目的位置、排列和尺寸。
  2. 默认模板: 提供多个设计精良的默认样式单元和布局方案，作为用户自定义的起点。
  3. 多角色切换: 提供一个横向滚动的角色选择栏来切换检视所有"在场" (`isPresent: true`) 角色的状态。
  4. 高级交互性:
      - 支持在样式单元的HTML模板中定义可点击元素。
      - 在"定义工作室"中，允许为条目关联"交互类型"。
      - 脚本运行时会根据定义，为可点击元素动态绑定相应事件。
  5. 折叠/展开: 每个由布局编排器创建的"区块"或"类别"应支持独立的折叠和展开，并记忆其状态。

## 管理器

- 功能: 系统的总配置中心和数据编辑器，以弹窗形式呈现。
- 核心要求: 包含以下五大功能模块，所有模块都需提供帮助指引（点击问号图标弹出浮窗）。务必追求高端、良好的现代化视图。

  1. 数据中心
      - 功能: 全功能的实时编辑器，用于增、删、改所有角色的状态数据。
      - UI:
        - 不同的第一层类型数据（例如用户可能自定义的第一层的"世界状态"（里面可能又有"场景信息"，"场景信息"下又有"时间地点"）、"角色状态"（里面可能又有"角色基本信息"，"角色基本信息"下又有"角色名字"等等））的编辑区应该有明确区分。
        - 编辑角色数据时，在角色名旁清晰地显示其唯一的 `char_id`。
        - 新增条目时，提供一个基于"定义工作室"中该分类下所有已定义`key`的智能推荐的输入框。
      - 交互: 实现标签双击编辑、带"x"按钮的标签删除、行级删除按钮、区级新增按钮等全功能交互。
      - 保存机制: 所有编辑在UI上即时生效（暂存模式），通过一个全局"保存"按钮一次性写入SST。
      - 提供UI控件（如眼睛图标）以手动切换角色的 `isPresent` 状态。
      - 未经定义的条目解析为Other类的条目。

  2. 数据工作室
      - 功能: 全局的"条目字典"，用于创建和管理所有数据和元数据的结构。支持导入/导出。
      - UI:
        - 分类列表: 允许用户增删改查数据类别，并设定其为"共享"或"角色"属性。
        - 条目列表: 展示所有已定义的条目，按类别分组。提供CRUD操作。
        - 状态栏属性编辑器:
          - `key`: 唯一键。
          - `name`: 显示名。
          - `category`: 从用户创建的分类列表中选择。
          - `icon`: 集成图标库（如Font Awesome/Lucide）的选择器，带搜索。
          - `entry_type`: 基础数据类型选择，只提供两个基础数据类型，直接存成更清晰易读的对象。
            - 单体 (`Single`)：用于描述一个单一实体。其所有`parts`共同构成一个不可分割的整体。
            - 列表 (`List`)：用于描述一组可独立增删的、结构相同的实体。
          - `parts`（`keys`, `labels`, `data_types`）: 根据`entry_type`动态生成。
            - 可增删，为每个结构指定名称`key`、UI文本标签`label`。
            - 可为每个结构指定基础类型`data_type`（文本、数字、布尔值）和验证规则（如数字范围，在后台生成Zod Schema，用于在AI更新数据时进行安全验证，防止AI返回错误类型或超出范围的数值，从而保护数据不被污染）。
          - `ui_type`: 下拉框，用于设置此条目的默认渲染样式，关联自"样式工坊"。
          - `interaction_type`: 下拉框，定义条目的交互行为。
          - `description`: 文本框，用户填写，用于指导AI。
          - `update_sample`: 文本框，根据以上信息自动生成一段示例的JSON Patch。

      - 功能按钮:
        - `注入世界书`: 将`description`和`update_sample`组合后，更新或创建到世界书中。

  3. 样式工坊
      - 功能: 创建可复用的"样式单元"和定制"全局主题"。存储在localStorage？支持导入/导出。
      - 样式单元:
        - 提供CRUD界面管理所有样式单元。
        - 提供内置的默认样式单元作为模板。
        - 编辑器提供"可视化GUI配置"和"手写代码配置"（填写HTML模板+局部CSS）。
        - HTML模板使用`{{placeholder}}`语法，占位符对应`parts`的`keys`（如`{{skill}}`）。可以有内置辅助占位符（如`{{progressbar}}`），但必须清晰说明（占位符参考）。
        - 提供一个带有可用类名和占位符参考的面板，必须是动态从CSS文件获取。
        - 必须提供一个实时预览区域，即时反馈样式修改效果。
      - 全局样式:
        - 提供一个CSS编辑器，允许用户定制状态栏容器、管理器弹窗、整体颜色主题等框架级元素的外观。（HTML模板的功能主要由布局编排器实现，但也要提供）
        - 同样提供可用CSS类名参考。（"可视化GUI配置"可能需要隐藏？）

  4. 布局编排器
      - 功能: 可视化地设计状态栏的布局。支持导入/导出。
      - UI:
        - 画布区域: 所见即所得的状态栏预览区。
        - 组件列表: 侧边栏，列出"定义工作室"中所有已定义的条目。
        - 元素检视: 侧边栏/浮窗，选中元素时支持精细调整参数。
      - 交互:
        - 支持从侧边栏拖拽条目到画布上。
        - 支持拖拽合并为行/列，自动均分布局。
        - 支持拖动调整手柄，手动设定宽高比例。
        - 所有布局调整实时预览。
      - 数据存储: 布局将被保存为一个JSON结构。

  5. 系统配置
      - 功能: 整合应用逻辑与系统级设置。
      - 子模块:
        - 原始数据: 显示目前/不同楼层的原始(json格式)状态栏数据给用户查阅甚至修改（类似`DevTools`？）。之后作为更复杂功能（例如操作撤销/重做（记录楼层数？））的入口。
        - 主题组合:
          - 功能: "主题应用者"，用于组合世界书、定义、样式和布局。支持导入/导出。
          - UI: 提供CRUD界面管理所有组合（如"魔法世界观"）。
          - 配置内容: 每个组合包含"条目列表"、"样式覆盖映射"和"布局选择"。
          - 交互: 在组合编辑器中，为每个条目提供"覆盖渲染样式"（覆盖该条目在其定义中设置的"默认样式"，默认选择"默认"）和"跳转到样式工坊/布局编排器的快捷按钮。
          - 应用逻辑: 提供"绑定到当前聊天"和"仅应用一次"功能。加载聊天时，脚本会自动应用绑定配置，覆写状态栏外观。
        - 叙事快照:
          - 提供开关以启用/禁用叙事生成器。
          - 允许用户创建和管理不同的叙事生成模板：提供CRUD操作；使用并提供`{{placeholder}}`；针对不同数据情况（共享数据、角色数据、User数据、`user_modified: true`的数据）和数据变化类型可以定制不同模板；提供一个编写提示词的文本框作为AI对数据变更叙事风格的总指导（例如望AI把npc的表现写成无视/神化/怀疑`user_modified: true`数据的变化）；支持导入/导出。
          - 显示快照元信息和功能按钮。提供开关以启用/禁用快照元信息注入世界书发送。
        - 条目管理：
          - 管理世界书条目，同样是通过一个"保存"按钮一次性写入。
        - 备份与迁移：
          - 提供可多选的导出成json文件功能、（拖入）导入json文件功能、恢复出厂设置（重置所有数据到初始状态）功能。
        - 使用指南：
          - 帮助文件，写出所有模块功能，引导用户使用。

## 后台逻辑

1. 数据:
    - 角色ID系统:
      - 必须实现角色唯一ID (`char_id`) 机制，确保数据不会因角色改名而丢失。角色数据以ID为键，ID格式为`char_{}`（用户角色固定ID为`char_user`），一般为AI生成，但也可由用户在数据中心添加角色时手动填写。
      - 在`chatVariables`中维护ID到最新角色名的映射表。
      - 所有UI渲染必须动态读取角色名。（除了用户角色`char_user`在SillyTavern中通过酒馆助手宏`{{substitudeMacros}}`获取真实persona名称）
    - 智能合并:
      - `source_id` 裁决: 每个数据项都有一个 `source_id` 属性，即它来源的消息楼层ID。只有当`Source.source_id >= Target.source_id`时，才允许更新。
      - 用户修改优先: 如果`Target`条目有一个 `user_modified: true` 的标志，必须无条件地阻止来自AI的任何更新。（这个现在感觉哪里不对，似乎不应该阻止来自AI的更新？初衷是发现AI可能会认为用户的修改是某种bug而修正，但现在应该可以通过快照元信息注入世界书发送，来对AI表明"这不是Bug，而是用户希望得到特殊体验而做的修改"？）
      - 优先级降级: 当用户发送新消息时，遍历整个SST，将所有`user_modified: true`的条目重置为`false`，并将其`source_id`更新为当前最新消息的ID，使其可以被后续的AI消息再次更新。
    - 会话完整性: 实现"时间线收缩"检测与警告。
      - SST中需包含一个元数据字段 `_meta.message_count`，用于记录上次更新时的总消息数。
      - 每次数据更新前，比较当前的总消息数与记录的 `message_count`。如果当前消息数小于记录值，意味着聊天记录被删除或修改。
      - 此时通过 `toastr.warning` 弹出一个警告通知，提示用户数据可能不一致，并建议用户通过管理器进行检查。
    - `nil`值 (临时隐藏指令) 处理: 在状态栏渲染时隐藏，在管理器中可见以让用户手动恢复，作为AI误删风险的安全机制。

2. AI交互:
    - 解析 JSON Patch (RFC 6902)来更新变量。
    - 叙事生成器: 检测数据变化并根据用户选择的风格模板生成叙事文本。（见系统配置里的叙事快照）

3. 脚本生命周期:
    - 必须实现基于唯一ID和全局信标的"单例所有权"机制，确保脚本热重载的稳定性。
      - 每个脚本实例在启动时生成一个唯一ID。
      - 使用 `top` 窗口上的一个全局对象来存储当前"所有者"实例的ID。
      - 新实例启动时，必须检查此全局对象。如果发现存在旧的"所有者"，新实例必须先调用一个全面的`cleanup`函数，彻底移除旧实例创建的所有UI元素（按钮、弹窗、样式）、事件监听器和定时器，然后再声明自己的所有权并执行初始化。

## UI要求

- 使用丰富的美学:设计应让用户第一眼就感到惊艳。使用现代网页设计的最佳实践(例如鲜艳的色彩、浅色深色模式切换、玻璃拟态和动态动画)来创造令人惊叹的第一印象。如果做不到这一点，是不可接受的。
- 优先考虑视觉卓越性:实施能让用户发出"WOW"惊叹并感觉极其高端的设计:
  - 避免使用通用颜色(纯红、蓝、绿)。使用精心挑选、和谐的调色板(例如HSL定制颜色、时尚的深色模式)。
  - 使用现代排版(例如Google Fonts中的 Inter、Roboto 或Outfit)，而不是浏览器默认字体。
  - 使用平滑的渐变，
  - 添加微妙的微交互动画以增强用户体验，
- 使用动态设计:感觉灵敏且生动的界面能鼓励交互。通过悬停效果和交互元素来实现这一点。特别是微交互动画，对于提高用户参与度非常有效。
- 高端设计。制作感觉高端且最先进的设计。避免创建简单的最小可行产品。
- 必须注意UI在不同屏幕大小下的舒适性，请将响应式设计纳入考虑。
- 尽可能不要使用emoji，而是使用美观、高级的ICON！例如Font Awesome/Lucide
- 除了美术上的用法，否则App的界面、按钮、文本，应当全部显示中文。

### 其他需注意的要求

- 样式(CSS):**绝对使用VanillaCSS以获得最大的灵活性和控制权**。避免使用TailwindCSS，除非用户明确要求;在这种情况下，请先确认要使用哪个 TailwindCSS 版本。
- CSS类名使用BEM命名规范，绝不使用 SillyTavern 的 CSS 变量，用独立命名（例如加上特别的前缀"omg-"）以防止污染。
- 除非用户要求，否则默认绝不使用占位符。
- 进行性能优化
  - 标题标签:为每个页面包含正确、描述性的标题标签。
  - 元描述:添加引人注目且准确总结页面内容的元描述。
  - 语义化 HTML:使用适当的HTML5语义元素。
  - 唯一ID:确保所有交互元素具有唯一的、描述性的ID，以便进行浏览器测试。
  - 性能:通过优化确保快速的页面加载时间。

## 补充附录

### 安全模式

必须实现一个全局键盘快捷键 (e.g. `Ctrl + Alt + Shift + R`)，用于在任何情况下（即使 UI 完全崩溃）强制触发安全模式，移除应用的样式并返回默认外观的应用，防止用户因样式错误而无法操作。

### 模块间深度链接

- 示例:
  1. 在定义编辑器中，样式选择器旁边必须有一个"编辑/创建样式"的快捷按钮。点击后，应自动切换到"样式工坊"模块，并打开样式编辑器（如果样式已存在，则加载它；如果不存在，则以当前定义的 `key` 作为预设打开新编辑器）。
  2. 在预设编辑器中，每个条目的样式覆盖下拉框旁也应有此快捷按钮。

### 模块功能补充与细化

#### 1. 数据中心

- 核心交互:
  - 清空数据必须为每个数据源（共享/单个角色）提供一个独立的"清空"按钮。只清空数据条目，不删除角色本身。
  - 删除角色: 角色列表中的非 `char_user` 角色必须提供删除按钮。删除操作会彻底移除该角色的所有数据、ID 映射和元数据。
- `Other` 分类的角色: `Other` 分类是作为所有未在"定义工坊"中找到对应规则的条目的回退容器 (Fallback Container)。解析器在遇到未知 `key` 时，应自动将其归入 `Other` 分类，确保数据不会丢失，并能在数据中心被用户看到和管理。

#### 2. 样式工坊

- 交互式实时预览:
  - 实现: 预览区域不仅仅是展示，它必须是可交互的。当用户点预览组件中的某个具体元素（如进度条、数值文本）时，右侧的"可视化 GUI配置"面板必须自动切换到该元素对应的 CSS 选择器，并加载其当前样式。

- 动态文档生成:
  - 规则: "可用 CSS 类名参考"功能绝不能硬编码。必须实现一个解析服务，保证文档实时性和准确性的核心。
  - 例如可以解析一种特定格式的注释：`/* @doc: Label | category: group | desc: Description */`，并将其后面的 CSS 选择器与注释内容关联，动态生成文档。
- 拖拽应用样式:
  - 交互: 用户必须能够直接从左侧的样式库中，将一个"样式单元"拖到右侧预览画布中的某个数据条目上。
  - 行为: 释放拖拽后，该数据条目的 `styleId` 必须立即在暂存数据中被更新，并在预览区实时反映出新的样式。
- 内置样式与用户样式:
  - 概念: 系统应附带一套设计精良的默认样式单元。这些样式是"只读"的，用户不能直接修改或删除。
  - 交互: 用户可以"查看"或"复制"这些默认样式，复制后会创建一个新的、可编辑的用户样式。这为用户提供了一个高质量的自定义起点。
- CSS 安全检查:
  - 在保存任何手写的 CSS 代码（尤其是在"全局主题"中）之前，必须进安全扫描，检测是否存在可能破坏整个UI的危险规则（如 `* {display: none; }`）。
  - 如果检测到，必须弹出一个措辞强烈的警告，让用户二次确认，并告知其如何使用"安全模式"恢复……或者直接禁止。
