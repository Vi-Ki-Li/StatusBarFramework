## 2) 本轮定稿（Discussion Freeze）

以下结论来自 2026-03-30 讨论，属于当前冻结口径：

1. 彻底移除“共享/角色”作为用户可见强制选择。
2. 首版采用“对象优先 + 集合同构优先”。
3. 首版允许嵌套到 2 层（未来可扩展）。
4. 背包等复杂容器采用“字典为主 + 可选顺序列表”。
5. 管理器增强优先于状态栏附加按钮（附加按钮仍是后续项）。
6. 主题组合需要补叙事模板绑定能力。
7. 导入不能长期停留在“仅覆盖”，需设计“覆盖/增量可选 + 冲突提示”。

## 3) 术语表（中英对照）

### 3.1 数据层级（Data Tier）

1. 全局数据层（Global Data Tier）
   - 当前聊天共享的一份数据，不会出现多份。
2. 模板定义层（Template Definition Tier）
   - 用户定义结构规则，不是实例值。
3. 实例数据层（Instance Data Tier）
   - 根据模板展开后的具体对象值。

补充说明（避免“全局数据是不是也在数据工作室”的混淆）：

1. 数据工作室管理的是“定义”（schema-like 规则），不是实例值本身。
2. 数据中心管理的是“值”（运行态数据），包含全局数据值与实例数据值。
3. 因此“全局数据”会在两处出现：
   - 在数据工作室：作为定义项的归属与组织；
   - 在数据中心：作为可编辑的真实值。
4. 两处职责不同：前者定义规则，后者编辑数据。

命名理由：

1. 避免“归属维度”抽象词导致的理解门槛。
2. 明确“模板定义层主要在数据工作室，实例数据层主要在数据中心/运行时”的出现位置。
3. 兼容后续扩展到角色/技能/装备/牌堆/道具等任意模板名，不硬编码业务词。

### 3.2 路径策略（Path Strategy）

1. 直接定位路径填写（Direct Target Path Entry）
   - 用户直接指定路径，适合唯一位置数据。
2. 根据模板展开填写（Template-Expanded Path Entry）
   - 先定义模板规则，再按实例自动展开路径。

命名理由：

1. 比“手工/自动”更直白，避免误会“只有一种是手工”。
2. 与工程语义“显式声明模板 + 隐式派生实例”兼容。

## 4) 数据契约草案（P0 Draft）

本草案为实现前契约，不是最终 schema；实现时必须遵守以下语义。

### 4.1 节点形态（Node Shape）

1. 标量（Scalar）
   - 单值：string/number/boolean。
2. 对象（Object）
   - 有字段名的结构体。
3. 集合（Collection）
   - 一组同类对象，首版以同构为主。

### 4.2 嵌套边界

1. 首版最大嵌套深度：2 层。
2. 超出边界的数据需标记并阻止直接写入主渲染链路（防止复杂度失控）。

### 4.3 集合策略

1. 首版默认同构集合（Homogeneous Collection）。
2. 异构集合（Heterogeneous Collection）列入后续阶段。
3. 集合成员身份优先 key/id，不依赖数组 index 作为主标识。

### 4.4 背包模型（范式样例）

1. 主存储：字典（Dictionary）。
2. 展示排序：可选顺序列表（Ordered View List）。
3. 顺序列表只负责 UI 展示顺序，不作为主真值源。

---

## 5) 强制约束（防偏离）

1. 禁止以“仅隐藏共享/角色 UI”代替模型重构。
2. 禁止在未通过契约门禁前直接推进对象/集合渲染改码。
3. 禁止使用硬编码业务模板名作为唯一支持路径（必须允许用户自定义模板名）。

---

## 6) 影响面清单（实现前必读）

1. 数据中心：聊天文件实时数据编辑入口与展示组织。
2. 数据工作室：模板定义与字段规则。
3. 渲染链路：对象/集合渲染策略。
4. 主题组合：补叙事模板绑定。
5. 导入导出：覆盖/增量模式与冲突提示。
6. 测试文档：术语替换与回归步骤更新。
