## DEC-001: 数据存储策略

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 酒馆助手提供多种变量存储（全局/角色卡/脚本/聊天/消息楼层），需要确定各类数据的存储位置。

**决定**:
| 数据 | 存储位置 | 原因 |
|------|----------|------|
| 定义/布局/配置/叙事模板 | 脚本变量 | 全局持久化，跨聊天共享 |
| 样式单元+全局主题 | localStorage | 可能体积较大 |
| 角色数据+ID映射 | 聊天变量 | 绑定到当前聊天 |
| 楼层快照 | 消息楼层变量 | 绑定到具体消息 |

## DEC-002: 是否依赖 MVU 变量框架

**日期**: 2026-03-16
**状态**: ✅ 已决定

**选项**:
- A: 强依赖 MVU，所有数据解析通过 MVU
- B: **独立实现数据解析，提供 MVU 兼容接口** ← 选择

**原因**: 用户希望框架能"独立行走"，适应不同变量系统（只要是 JSON 存储）。

**实现方式**:
- 自建 JSON Patch 解析器
- 提供适配层接口，可挂接 MVU 的 `parseMessage` / `VARIABLE_UPDATE_ENDED` 等事件
- 当检测到 MVU 存在时，自动启用兼容模式

## DEC-003: user_modified 行为

**日期**: 2026-03-16
**状态**: ✅ 已决定

**选项**:
- A: 阻止 AI 更新被标记的条目
- B: **不阻止更新，通过叙事快照告知 AI "这是用户的主动修改"** ← 选择

**原因**: 阻止更新过于武断；通过提示词让 AI 理解用户意图更灵活。

## DEC-004: CSS 框架选择

**日期**: 2026-03-16
**状态**: ✅ 已决定

**决定**: **全部使用 Vanilla CSS**
- BEM 命名规范
- `omg-` 前缀防止污染酒馆样式
- 不使用酒馆的 CSS 变量
- 不使用 TailwindCSS

**原因**: 用户希望状态栏和管理器的 UI 都能由用户自定义。Vanilla CSS 提供最大的可控性和透明度，用户手写 CSS 时不需要理解 Tailwind 的工具类体系。

## DEC-006: 管理器样式隔离

**日期**: 2026-03-16
**状态**: ✅ 已决定

**决定**: 管理器浮窗使用 `createScriptIdIframe` 在独立 iframe 中渲染，与酒馆网页样式完全隔离。

**原因**:
- 管理器有大量自定义 UI，不应受酒馆样式干扰
- 用户自定义的全局主题也不应影响酒馆
- iframe 隔离是脚本规则中推荐的方案
