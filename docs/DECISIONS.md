## DEC-001: 数据存储策略

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 酒馆助手提供多种变量存储（全局/角色卡/脚本/聊天/消息楼层），需要确定各类数据的存储位置。

**决定**:
| 数据                    | 存储位置     | 原因                                   |
| ----------------------- | ------------ | -------------------------------------- |
| 定义/布局/配置/叙事模板 | IndexedDB    | 跨聊天持久化，体积可能较大，无大小限制 |
| 样式单元+全局主题       | IndexedDB    | 包含HTML/CSS模板，体积可能较大         |
| 角色数据+ID映射         | 聊天变量     | 绑定到当前聊天                         |
| 楼层快照                | 消息楼层变量 | 绑定到具体消息                         |
| 系统配置(少量)          | 脚本变量     | 少量简单设置，如启用/禁用开关          |

**不使用脚本变量存储大量配置数据的原因**: 脚本变量本质存储在酒馆的 JSON 文件中，不适合存储大量 HTML/CSS 模板、布局 JSON 树等数据。

**IndexedDB 的权衡**: 浏览器本地存储，换浏览器/清缓存会丢失——通过导入/导出功能解决。

## DEC-002: 是否依赖 MVU 变量框架

**日期**: 2026-03-16
**状态**: ✅ 已决定

**选项**:
- A: 强依赖 MVU，所有数据解析通过 MVU
- B: **独立实现数据解析，提供 MVU 兼容接口** ← 选择

**原因**: 用户希望框架能"独立行走"，适应不同变量系统（只要是 JSON 存储）。

**实现方式**:
- 自建 JSON Patch 解析器
- 提供适配层接口，可挂接 MVU 的 `parseMessage` / `VARIABLE_UPDATE_ENDED` 等事件
- 当检测到 MVU 存在时，自动启用兼容模式

## DEC-004: CSS 框架选择

**日期**: 2026-03-16
**状态**: ✅ 已决定

**决定**: **全部使用 Vanilla CSS**
- BEM 命名规范
- `omg-` 前缀防止污染酒馆样式
- 不使用酒馆的 CSS 变量
- 不使用 TailwindCSS

**原因**: 用户希望状态栏和管理器的 UI 都能由用户自定义。Vanilla CSS 提供最大的可控性和透明度，用户手写 CSS 时不需要理解 Tailwind 的工具类体系。

## DEC-007: 数据工作室重新定位 — 移除 entry_type/parts

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 原设计中 `entry_type` (Single/List) 和 `parts` (keys/labels/data_types) 是为自定义更新指令设计的——需要告诉解析器数据结构。但改用 JSON Patch (RFC 6902) 后，解析器直接操作 JSON 路径，不需要预先知道数据结构。

**选项**:
- A: 保留 entry_type/parts 体系，同时也用 JSON Patch
- B: **移除 entry_type/parts，将数据工作室重新定位为"显示/验证元数据"** ← 选择

**决定**: 数据工作室不再定义数据结构（JSON Patch 自动处理），而是定义数据的**显示方式**和**验证规则**。

**保留的属性**:
| 属性                   | 用途                                            |
| ---------------------- | ----------------------------------------------- |
| `key`                  | JSON 路径标识 + 显示键                          |
| `name`                 | 显示名称                                        |
| `category`             | 分类（按数据层级组织，不作为共享/角色强制入口） |
| `icon`                 | 图标                                            |
| `data_type` + 验证规则 | Zod 验证（防止AI写错类型）                      |
| `ui_type`              | 关联样式工坊的渲染样式                          |
| `interaction_type`     | 关联交互行为                                    |
| `description`          | AI 指导文本                                     |
| `update_sample`        | JSON Patch 示例                                 |

**移除的属性**:
| 属性                             | 原因                      |
| -------------------------------- | ------------------------- |
| `entry_type` (Single/List)       | JSON Patch 不需要预知结构 |
| `parts` (keys/labels/data_types) | 同上                      |

**核心转变**: 数据的实际结构完全由用户的 JSON 数据本身定义。框架遇到未在数据工作室定义的 key 时，自动用 `Other` 回退显示。

## DEC-008: 布局编排器 — 分层布局系统取代纯行列排布

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 原设计只支持"拖拽合并为行/列，自动均分"，本质是简单 flexbox，无法实现瀑布流、网格布局、自由定位等。

**决定**: 采用**分层布局系统**，支持多种 CSS 布局模式。

**三层架构**:

```
层级 1: 布局模式 (Layout Mode)
  用户为每个容器选择布局策略:
  ├─ 流式 (Flexbox) — 行/列/换行，flexbox 参数可调
  ├─ 网格 (CSS Grid) — 行列定义+区域命名
  ├─ 瀑布 (Masonry) — 自动填充瀑布流
  ├─ 自由 (Absolute) — 自由定位+叠加
  └─ 自定义 (Custom CSS) — 用户直接写 CSS

层级 2: 容器嵌套 (Containers)
  容器可以嵌套，每个容器有自己的布局模式
  形成树状结构

层级 3: 条目放置 (Items)
  将数据工作室条目拖入容器
  每个条目可调: 尺寸/间距/对齐/跨度
```

**编辑器交互方式**:
- 初学者: 选择布局模式 → 拖入条目 → 检视面板调参 → 实时预览
- 高级用户: 直接编辑布局 JSON 或写 Custom CSS

**布局数据格式**: 树状 JSON 结构，每个节点包含 `type`(container/item)、`mode`、布局参数、`children`。

## DEC-009: 文档工作流与职责固定化

**日期**: 2026-03-19
**状态**: ✅ 已决定

**背景**: 随迭代增长，`STATE.md` 混入大量历史，`phases/` 缺少阶段文档，导致新会话恢复成本高、文档职责混乱。

**决定**:

1. 固定流程为：**需求 → 讨论 → 研究 → 规划 → 执行 → 验证**
2. 文档职责明确：
   - `STATE.md` 只保留当前快照
   - `phases/phase-N-*.md` 记录阶段完整过程
   - `DECISIONS.md` 记录关键取舍
   - `TESTING.md` 持续补齐可复用测试
   - `ARCHITECTURE.md` 维护系统结构总览
3. 新会话按固定顺序读取：STATE → WORKFLOW → ARCHITECTURE → DECISIONS → ROADMAP → 最新 phase 文档

**原因**:

- 降低上下文腐化
- 提高跨会话可恢复性
- 让非开发用户也能快速理解当前状态与下一步

## DEC-011: UI 验证判定口径（截图优先，代码辅助）

**日期**: 2026-03-20
**状态**: ✅ 已决定

**背景**: UI 回归中存在“代码显示有入口，但用户截图不可见”的冲突；同时逐步截图容易产生重复证据。

**决定**:

1. 结论判定采用“截图优先”：截图不可见/不可操作，即判定用户侧不可见/不可操作。
2. 代码仅用于辅助解释和定位根因，不可覆盖截图结论。
3. 证据策略采用“before 分级 + after 必拍”：
  - `after` 每步必拍；
  - `before` 在关键步骤必拍，其余允许复用上一张 `after` 作为引用，降低重复截图。

**原因**:

- 避免“工程视角可见”与“用户视角不可见”混淆；
- 保证证据链可审计，同时降低截图噪声与执行成本。

## DEC-012: 主题作用域渲染与布局分组元字段

**日期**: 2026-03-26
**状态**: ✅ 已决定

**背景**:

- 复刻主题在紧凑布局下出现“未布局条目”，原因是渲染器会遍历全部 definitions。
- 布局树中的 container `label` 默认不会渲染，导致“分类标题不可见”。
- 同 ID 样式更新后可能继续显示旧效果，造成“样式未应用”的体感问题。

**决定**:

1. 有激活主题时，渲染范围按 `theme.entryIds` 过滤。
2. 为布局节点引入可选元字段（渲染器识别）：
  - `showLabel: boolean`
  - `collapsible: boolean`
3. 每次加载定义时重置动态样式缓存，确保同 ID 样式热更新可见。

**原因**:

- 让 theme 真正成为“渲染边界”，避免非主题条目污染布局结果。
- 在不破坏现有 JSON 契约的前提下提供“分类容器化”的轻量能力。
- 减少导入调试成本，提升“改即见效”的可预期性。

**兼容性**:

- `showLabel/collapsible` 为可选字段，不配置时保持旧行为。
- 现有布局 JSON 与主题 JSON 无需迁移即可继续使用。

## DEC-013: 分类动态绑定与折叠状态持久

**日期**: 2026-03-26
**状态**: ✅ 已决定

**背景**:

- 用户希望“数据工作室分类新增条目后，布局容器自动跟随”，但不是所有容器都要强制启用该行为。
- 复刻面板中出现“分组折叠后概率性不展开”与“折叠后高度未回收”的体感问题。

**决定**:

1. 布局容器支持可选动态绑定字段：`autoSyncFromCategory: boolean`、`bindCategoryId: string`。
1. 动态绑定模式提供排序策略开关：`categorySortStrategy: 'category' | 'layout'`；`category` 按分类原始顺序，`layout` 按布局手动顺序优先并将新增条目追加到末尾。
1. 分组折叠状态改为脚本内存持久（按 `node.id` 记录），避免重渲染导致点击状态丢失。
1. 折叠态强制覆盖 `min-height/height/padding/gap`，确保高度可回收。

**原因**:

- 动态绑定满足“定义先行”的可维护性，且通过开关保证兼容不同工作流。
- 排序策略开关平衡“自动跟随”与“人工编排”的控制需求。
- 折叠状态持久可降低事件竞态带来的概率性异常。

**兼容性**:

- 新字段均为可选，不影响旧布局 JSON。
- 未开启动态绑定的容器保持原有静态 children 渲染。

## DEC-014: P0 术语统一与入口去耦

**日期**: 2026-03-30
**状态**: ✅ 已决定

**背景**:

- “共享/角色”在历史实现中同时承担了数据归属、UI入口、测试口径三重含义，导致非专业用户理解成本高。
- 项目已确定对象优先与模板驱动方向，继续保留“共享/角色强制入口”会制造模型和UI不一致。

**决定**:

1. 术语统一为“数据层级（Data Tier）”：全局数据层、模板定义层、实例数据层。
2. 路径策略统一为“直接定位路径填写 / 根据模板展开填写”。
3. 从 P0 开始，需求与测试文档不再将“共享/角色”作为用户强制入口口径。
4. 明确禁止“仅隐藏共享/角色 UI”替代模型重构。

**原因**:

- 降低术语歧义，提升非专业用户的可理解性。
- 让文档、测试、实现目标在 Phase 8.13 之后保持同一口径。

**兼容性**:

- 历史文档中的“共享/角色”描述保留为历史背景，不作为新门禁标准。
- 运行期兼容策略在实现阶段单独定义，不在本决策中展开。
