## DEC-001: 数据存储策略

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 酒馆助手提供多种变量存储（全局/角色卡/脚本/聊天/消息楼层），需要确定各类数据的存储位置。

**决定**:
| 数据                    | 存储位置     | 原因                                       |
| ----------------------- | ------------ | ------------------------------------------ |
| 定义/布局/配置/叙事模板 | IndexedDB    | 跨聊天持久化，体积可能较大，无大小限制     |
| 样式单元+全局主题       | IndexedDB    | 包含HTML/CSS模板，体积可能较大             |
| 角色数据+ID映射         | 聊天变量     | 绑定到当前聊天                             |
| 楼层快照                | 消息楼层变量 | 绑定到具体消息                             |
| 系统配置(少量)          | 脚本变量     | 少量简单设置，如启用/禁用开关              |

**不使用脚本变量存储大量配置数据的原因**: 脚本变量本质存储在酒馆的 JSON 文件中，不适合存储大量 HTML/CSS 模板、布局 JSON 树等数据。

**IndexedDB 的权衡**: 浏览器本地存储，换浏览器/清缓存会丢失——通过导入/导出功能解决。

## DEC-002: 是否依赖 MVU 变量框架

**日期**: 2026-03-16
**状态**: ✅ 已决定

**选项**:
- A: 强依赖 MVU，所有数据解析通过 MVU
- B: **独立实现数据解析，提供 MVU 兼容接口** ← 选择

**原因**: 用户希望框架能"独立行走"，适应不同变量系统（只要是 JSON 存储）。

**实现方式**:
- 自建 JSON Patch 解析器
- 提供适配层接口，可挂接 MVU 的 `parseMessage` / `VARIABLE_UPDATE_ENDED` 等事件
- 当检测到 MVU 存在时，自动启用兼容模式

## DEC-004: CSS 框架选择

**日期**: 2026-03-16
**状态**: ✅ 已决定

**决定**: **全部使用 Vanilla CSS**
- BEM 命名规范
- `omg-` 前缀防止污染酒馆样式
- 不使用酒馆的 CSS 变量
- 不使用 TailwindCSS

**原因**: 用户希望状态栏和管理器的 UI 都能由用户自定义。Vanilla CSS 提供最大的可控性和透明度，用户手写 CSS 时不需要理解 Tailwind 的工具类体系。

## DEC-006: 管理器样式隔离

**日期**: 2026-03-16
**状态**: ✅ 已决定

**决定**: 管理器浮窗使用 `createScriptIdIframe` 在独立 iframe 中渲染，与酒馆网页样式完全隔离。

**原因**:
- 管理器有大量自定义 UI，不应受酒馆样式干扰
- 用户自定义的全局主题也不应影响酒馆
- iframe 隔离是脚本规则中推荐的方案

## DEC-007: 数据工作室重新定位 — 移除 entry_type/parts

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 原设计中 `entry_type` (Single/List) 和 `parts` (keys/labels/data_types) 是为自定义更新指令设计的——需要告诉解析器数据结构。但改用 JSON Patch (RFC 6902) 后，解析器直接操作 JSON 路径，不需要预先知道数据结构。

**选项**:
- A: 保留 entry_type/parts 体系，同时也用 JSON Patch
- B: **移除 entry_type/parts，将数据工作室重新定位为"显示/验证元数据"** ← 选择

**决定**: 数据工作室不再定义数据结构（JSON Patch 自动处理），而是定义数据的**显示方式**和**验证规则**。

**保留的属性**:
| 属性                   | 用途                       |
| ---------------------- | -------------------------- |
| `key`                  | JSON 路径标识 + 显示键     |
| `name`                 | 显示名称                   |
| `category`             | 分类（共享/角色）          |
| `icon`                 | 图标                       |
| `data_type` + 验证规则 | Zod 验证（防止AI写错类型） |
| `ui_type`              | 关联样式工坊的渲染样式     |
| `interaction_type`     | 关联交互行为               |
| `description`          | AI 指导文本                |
| `update_sample`        | JSON Patch 示例            |

**移除的属性**:
| 属性                             | 原因                      |
| -------------------------------- | ------------------------- |
| `entry_type` (Single/List)       | JSON Patch 不需要预知结构 |
| `parts` (keys/labels/data_types) | 同上                      |

**核心转变**: 数据的实际结构完全由用户的 JSON 数据本身定义。框架遇到未在数据工作室定义的 key 时，自动用 `Other` 回退显示。

## DEC-008: 布局编排器 — 分层布局系统取代纯行列排布

**日期**: 2026-03-16
**状态**: ✅ 已决定

**背景**: 原设计只支持"拖拽合并为行/列，自动均分"，本质是简单 flexbox，无法实现瀑布流、网格布局、自由定位等。

**决定**: 采用**分层布局系统**，支持多种 CSS 布局模式。

**三层架构**:

```
层级 1: 布局模式 (Layout Mode)
  用户为每个容器选择布局策略:
  ├─ 流式 (Flexbox) — 行/列/换行，flexbox 参数可调
  ├─ 网格 (CSS Grid) — 行列定义+区域命名
  ├─ 瀑布 (Masonry) — 自动填充瀑布流
  ├─ 自由 (Absolute) — 自由定位+叠加
  └─ 自定义 (Custom CSS) — 用户直接写 CSS

层级 2: 容器嵌套 (Containers)
  容器可以嵌套，每个容器有自己的布局模式
  形成树状结构

层级 3: 条目放置 (Items)
  将数据工作室条目拖入容器
  每个条目可调: 尺寸/间距/对齐/跨度
```

**编辑器交互方式**:
- 初学者: 选择布局模式 → 拖入条目 → 检视面板调参 → 实时预览
- 高级用户: 直接编辑布局 JSON 或写 Custom CSS

**布局数据格式**: 树状 JSON 结构，每个节点包含 `type`(container/item)、`mode`、布局参数、`children`。
