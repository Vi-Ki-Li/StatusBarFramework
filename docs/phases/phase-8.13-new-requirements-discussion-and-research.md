# Phase 8.13：新增需求对齐与 P0 开工门禁

日期：2026-03-26（原始立项）/ 2026-03-30（本轮对齐补充）

## 1) 目标（Goal）

本阶段不写业务代码，目标是把“对象优先 + 模板驱动 + 去共享/角色硬分类”的 P0 口径一次对齐，作为后续实现和回归的强制门禁。

原始新增需求归档：`docs/REQUIREMENTS-RAW.md` 中“2026-03-26 新增补充需求”。

## 2) 本轮已对齐结论（2026-03-30）

### P0（必须先落）

1. 彻底移除“共享/角色”作为用户可见的强制选项。
2. 首版采用“对象优先 + 集合同构优先”的数据表达。
3. 首版允许嵌套到 2 层（未来可扩展更深）。
4. “背包”这类容器采用“字典为主 + 可选顺序列表”方案。

### P1（P0 后推进）

1. 管理器体验增强（记忆页面、位置与尺寸）。
2. 主题组合补叙事模板绑定能力（当前属于遗漏点）。
3. 导入策略从“仅覆盖”扩展为“覆盖/增量可选”，并配冲突提示。

### 明确排除（避免偏离）

1. 禁止把“移除共享/角色”降级成“仅隐藏 UI 文案”。
2. 必须同步处理数据模型、导入导出、测试口径与术语口径，不得只改一个层面。

## 3) 术语定稿（中英对照）

为降低非专业用户理解成本，本阶段将“数据归属维度”重命名为“数据层级（Data Tier）”，避免“都像数据工作室选项”的误解。

### 3.1 数据层级（Data Tier）

1. 全局数据层（Global Data Tier）
   - 含义：当前聊天共享的一份数据。
   - 常见位置：数据中心可编辑，渲染时全局可读。
2. 模板定义层（Template Definition Tier）
   - 含义：用户定义“结构规则”的层，不是具体实例值。
   - 常见位置：数据工作室。
3. 实例数据层（Instance Data Tier）
   - 含义：根据模板展开后的具体对象值。
   - 常见位置：数据中心与运行时。

说明：
1. 旧“共享”更接近“全局数据层 + 单点写入”。
2. 旧“角色”更接近“模板定义层（模板名=角色）+ 实例数据层（每个角色实例）”。

### 3.2 路径策略（Path Strategy）

为避免“手工/自动”歧义，本阶段采用更直白命名：

1. 直接定位路径填写（Direct Target Path Entry）
   - 含义：用户直接指定这条数据写到哪。
   - 适用：一次性、唯一位置的数据。
2. 根据模板展开填写（Template-Expanded Path Entry）
   - 含义：用户先定义模板，脚本按实例自动展开具体路径。
   - 适用：角色、技能、装备、道具等可重复对象。

补充说明：
1. 两种策略都可能有“手工创建”环节，区别在“后续是否由模板规则自动展开实例路径”。
2. 上述命名与“显式声明模板、隐式派生实例”的工程表述兼容。

## 4) 结构策略（P0 约束）

1. 集合首版优先同构（Homogeneous Collection First）。
2. 异构集合（Heterogeneous Collection）不作为首版必达项。
3. 背包/牌堆等复杂容器：
   - 主真值结构：字典（Dictionary / Map-like Object）；
   - 可选显示结构：顺序列表（Ordered View List）。
4. 不把数组下标作为主身份标识；优先稳定 key/id。

## 5) 已识别的现状缺口（本阶段仅记录）

1. 主题组合当前未绑定叙事模板（Narratives）能力，需要在后续补齐。
2. 导入当前以覆盖语义为主，缺少“增量导入 + 冲突提示 + 回滚说明”。

## 6) 实施切片（执行顺序）

1. 批次 A（P0）：
   - 完成 P0 契约文档与门禁检查；
   - 确立去“共享/角色”后的统一术语与测试口径；
   - 明确对象优先、同构优先、2 层嵌套的边界。
2. 批次 B（P1）：
   - 管理器体验增强（页面记忆、位置大小）；
   - 主题组合补叙事模板；
   - 导入模式扩展为覆盖/增量可选。

P0 开工门禁与契约草案：`docs/P0-PRE-IMPLEMENTATION-ACCEPTANCE-CHECKLIST.md`

未通过门禁前，不进入 P0 代码实现。
