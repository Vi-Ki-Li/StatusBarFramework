# Phase 8.2（批次 B）：UI 验证流程规范化与分模块 UX 收敛（待确认）

## 1) 需求（Requirement）

- 维持“截图优先结论”，但允许“代码辅助理解”：
  - 不允许因为代码里有就判定用户可见；
  - 若截图显示不可见/不可用，以截图事实为准，再用代码定位原因。
- 产出完整截图清单与问题分级（已完成首轮）。
- 修订截图策略，减少重复截图噪声，同时保留可审计对比证据。
- 进入下一阶段：
  - 需求 → 讨论 → 研究 → 规划；
  - 用户确认后再执行修复与验证。
- 规划后续“子模块流程测试”规范：
  - click/脚本双路径；
  - 零门槛可用性（新用户是否知道下一步）；
  - 截图留证模板。

## 2) 讨论（Discussion）

### 本批范围（确认后执行）

- 优先解决桌面端分模块可用性与新手可理解性问题（可复现、可留证）。
- 建立“截图优先 + 代码辅助”的统一判定口径。
- 在测试规范里引入截图降噪策略，减少重复图。
- 输出后续子模块测试规范草案（先文档，不立即大规模执行）。
- 先注入标准测试数据，放大问题可见性，避免“看起来没问题但无法稳定复现”。

### 不在本批

- 不直接推进全部模块的深层交互修复。
- 不处理与入口阻塞无关的历史问题。
- 移动端“管理器不显示”暂不作为本批强制修复项（响应式全面适配属于 Phase 10 范畴）。

## 3) 研究（Research）

- 桌面端：五模块和系统配置子模块可达，已有完整证据链。
- 移动端：在 360x780 下，“打开管理器”点击后多次 before/after 无明显状态变化；该问题暂归档为响应式适配相关风险，待 Phase 10 系统处理。
- 截图重复的主要来源：
  - 某些点击后页面无可见变化；
  - iframe 重建导致重复回到起始态；
  - 对每一步都强制 before，产生高相似度图片。
- `[酒馆助手]连接实时监听器出错，尝试重连...` 在本次不作为 UI 设计缺陷定性（已由用户说明与 `pnpm watch` 未运行相关），后续仅作为环境状态记录。

### Phase 0-8 缺口（可能影响 UX 体验）

- Phase 2：`ui_type` 关联样式工坊未完成。
- Phase 3：布局 JSON → DOM 渲染仍是待项（当前默认垂直布局）。
- Phase 4：可视化 GUI 配置面板、拖拽应用样式未完成。
- Phase 5：WYSIWYG 画布与调整手柄未完成。
- Phase 8：世界书条目管理未完成。

## 4) 规划（Planning）

- [ ] P0 修复方案讨论稿：桌面端分模块关键流程阻塞（优先级最高）
- [ ] 入口修复前置验证模板：必须先确认 `pnpm watch` 运行，再进行 UI 结论判定
- [ ] 截图策略升级（减少重复）：
  - `before` 必拍：首次进入模块、即将触发状态变化、阻塞留证；
  - `before` 可复用：若“上一张 after 与当前操作前状态完全一致”，允许将上一张 after 作为当前 before 引用；
  - `after` 必拍：每次点击后的结果图必须保留。
- [ ] 子模块测试规范草案：click/脚本双路径 + 零门槛可用性检查项 + 证据命名模板
